# Adjutorix Guided Product Experience

Adjutorix contains a broad governed execution, verification, replay, evidence,
release, publication, and distribution authority stack.

The product experience must not expose that stack as dozens of simultaneous
floating panels.

## Product rule

The operator chooses an outcome:

1. Understand
2. Plan
3. Build
4. Verify
5. Ship

Adjutorix then exposes the most relevant capability while every other authority
surface remains mounted, preserved, and inactive.

## Primary tools

The guided product shell treats these as first-class tools:

- AI Workbench Bridge
- Workspace Context
- AI Patch Runway
- Patch Verify

The historical and current authority-chain surfaces remain searchable through
the same shell.

## Interaction contract

- No uncontrolled overlay fanout.
- Zero managed surfaces visible by default.
- Exactly one managed surface may be active.
- Escape returns to the primary workspace.
- Command-K opens capability search.
- Command-Shift-P toggles the full power deck.
- Every authority component remains mounted.
- Existing confirmation, governance, replay, verification, and evidence
  boundaries remain unchanged.

## Competitive product position

Adjutorix combines:

- mode-based task navigation;
- persistent workspace context;
- code search and editing;
- governed patch execution;
- explicit diff and verification control;
- deterministic replay;
- evidence and receipt generation;
- finality and publication authority;
- a single progressive-disclosure product experience.

The objective is not to imitate another editor. The objective is to make a
more complete system substantially easier to understand and operate.

## Semantic primary-tool identity

Primary product tools are classified by product role rather than by one
hardcoded implementation identifier.

The guided shell resolves each mounted tool through its canonical identifier,
visible heading, accessibility label, title attribute, and stable semantic
keywords. This permits the live AI Workbench Bridge to remain a primary
Understand/Chat tool even when its implementation-specific DOM identifier
changes.

The four required primary roles remain:

- provider bridge;
- workspace context;
- patch runway;
- patch verification.

## Constrained provider topology fallback

The guided shell first resolves primary tools through canonical identifiers and
semantic product identity.

When those methods leave exactly the integrated assistant unclassified, the
remaining non-runway `adjutorix-ai-*` product surface is assigned the
`providerBridge` role.

The fallback explicitly excludes every `adjutorix-ai-runway-*` surface, so the
95-surface governance, evidence, finality, archive, publication, and
distribution authority chain can never be reclassified as the conversational
provider.
